# AI2 PHP SDK 架构设计

## 1. 目标

本 SDK 的第一优先级不是“快速堆接口”，而是先锁定一套可长期演进的公开结构。

必须满足：

- 对外调用方式稳定
- 内部请求调度层可重构
- `merchant` 与 `system` 两套鉴权严格隔离
- 资源按业务对象组织，不按 URL 零散暴露
- 后续新增接口时，尽量只新增资源方法，不改现有层级

## 2. 公开调用层

公开入口固定为命名空间函数：

```php
\TaohuaHub\Ai2\merchant($config);
\TaohuaHub\Ai2\system($config);
```

公开调用风格固定为：

```php
$merchant = \TaohuaHub\Ai2\merchant($config);

$merchant->sessions()->create($data);

$session = $merchant->session($sessionId);
$session->get();
$session->messages()->getList($query);
$session->messages()->get($messageId);
$session->messages()->send($data);
$merchant->messages()->get($messageId);
$session->jobs()->getList($query);
$session->stats();
```

## 3. 分层约束

### Public 层

这些命名空间视为稳定公开 API：

- `TaohuaHub\Ai2\Config\*`
- `TaohuaHub\Ai2\Client\*`
- `TaohuaHub\Ai2\Resource\*`
- `TaohuaHub\Ai2\Context\*`
- `TaohuaHub\Ai2\Exception\*`
- `TaohuaHub\Ai2\Result\*`
- `TaohuaHub\Ai2\Contract\*`
- `TaohuaHub\Ai2\merchant()`
- `TaohuaHub\Ai2\system()`

### Internal 层

这些命名空间不对外承诺兼容：

- `TaohuaHub\Ai2\Internal\*`
- `TaohuaHub\Ai2\Support\*`

后续重构默认只允许动 Internal 层。

## 4. 核心职责

### Client

负责进入业务域：

- `MerchantClient`
- `SystemClient`

### Resource

负责集合级资源：

- `sessions()`
- `jobs()`
- `stats()`
- `merchants()`
- `models()`

### Context

负责单实例上下文：

- `session($sessionId)`
- `session(...)->messages()`

### Dispatcher

内部唯一请求调度核心：

- 构建 URL
- 组装 query / body
- 生成请求头
- 商户签名
- 发起 HTTP 请求
- 解析统一响应 `ret/msg/data`
- 将业务错误转换成 SDK 异常

## 5. 为什么不用一个大 Client

如果把所有接口都直接堆到一个 Client 上：

- 层级语义会很差
- 会话和消息关系不清晰
- 长期会退化成方法垃圾场

因此采用：

- `client -> resource -> context`

而不是：

- `client -> 所有方法`

## 6. 当前决定

- 不引入总门面类 `Ai2`
- 入口直接使用命名空间函数
- `sessions()->getList()` 作为规划接口保留，但当前服务端尚未提供会话列表接口
- `session($id)->messages()->send()` 是主风格
- 后续若保留 `messages()` 根资源，也只作为补充入口，不替代上下文风格
